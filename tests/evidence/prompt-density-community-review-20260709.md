# 2026-07-09 Prompt 密度与社区路径只读审查

## 背景

连续多轮弱模型 prompt 候选回退后，本轮按“先审查信息熵和重复表达，再决定是否最小压缩”的原则推进。目标不是继续往 `SKILL.md` 堆规则，而是判断是否可以通过入口瘦身和渐进式路由降低弱模型注意力负担，同时不损害强模型写稿质量。

## 本地信息密度审查

本地 `rg` 扫描显示，常驻入口中超过 220 字的高密度行集中在：

- `SKILL.md` description：用于触发路由，暂不压缩。
- `SKILL.md` 压缩、顺稿、排版交付、日期/反馈要求、字段式材料、脚本说明等长行。
- `workflow.md:95` 的事实充分性软处理极长，但它属于按需 reference，不是默认常驻入口。

重复表达主要集中在：

- 事实不足、正文外待确认、不编造强判断。
- Markdown 残留。
- 最新版底稿和旧稿不回流。
- Word/docx 正式交付前要素核对。

判断：可审查的热点存在，但不能简单删入口锚点。此前真实写稿 A/B 已证明，仅拆短入口句不能稳定改善弱模型，反而可能带来事实关系错误。

## 独立 Subagent 审查

只读 subagent 对 `SKILL.md` 与 `references/*.md` 的结论：

- 入口确实有事实不足、排版/Word、只审不改、结构动作和字段边界等重复表达。
- 但这些多数是历史弱模型失败后形成的护栏，不能直接删除。
- 较稳方向是保留入口短原则，把细例和展开说明放回 `workflow.md`、`format-gbt9704.md`、`review-checklist.md`、`anti-ai-patterns.md`。
- 不应压缩联网边界、当前日期/落款规则、字段式材料硬边界和 reference 渐进读取表。

## 社区检索

使用 SkillHub CLI 做只读检索：

- `skillhub --skip-self-upgrade search 公文 --search-limit 8 --json`
- `skillhub --skip-self-upgrade search 降AI味 --search-limit 8 --json`
- `skillhub --skip-self-upgrade search 审稿 --search-limit 8 --json`

代表性参考下载到 `output/skillhub-reference-20260709/`，仅用于只读检查：

- `official-document-skill`：单文件大入口、内置评分、人民日报风格和大量文种模板。可借鉴“事实密度、判断强度、段落功能”思路；不借鉴单文件膨胀、评分、人民日报强风格和模板化正文。
- `plan-proofread`：入口列模式和 references 分层，审稿输出按位置、类型、严重度、建议组织。可借鉴“入口只放索引，细则进 reference”和“审稿清单化”；不借鉴重脚本链、质量评分和长报告审校专用流程。
- `unclecheng-reduce-ai-perception-v2`：有大量禁词、口语化、活人感、硬性替换和多轮改写流程。可借鉴“只改怎么说，不改说什么”的保真原则；不借鉴硬禁词、口语化、第一人称、情绪化、标点禁令和完整三遍改写。

ClawHub、aiskill.market、mcpmarket.com 和 GitHub 的公开网页搜索未取得可直接复核的同类源码候选；不能把无关搜索结果包装成参考证据。

## 入口瘦身候选结果

尝试过一个更激进的候选：把 `SKILL.md` 的“起草或改写”事实边界段改成“短原则 + 转读 `workflow.md` / `handling-elements.md`”。候选同步到各镜像后，定向单测立即失败：

- `python -m unittest tests.test_real_prompt_ablation tests.test_skill_boundary`
- 结果：41 tests 中 6 个失败。

失败原因不是环境问题，而是现有回归测试要求入口常驻保留关键锚点，例如：

- `用户明示某些事项未提供`
- `不要做调查问卷式问题清单`
- `不把上一轮待确认事项升级为阻断条件`
- `材料未给风险结论`
- `未发现重大隐患`
- `总体较好`
- `能够正常开展`
- `不得为显得完整而补造未提供的牵头部门、责任部门、管理动作、整改动作、成果总结或跟踪督办安排`

如果为了让测试通过把这些锚点全部补回，候选会退化为上一轮“拆行/换行”形态；该形态已在真实 A/B 中因弱模型事实关系回退被否定。因此本轮不继续修这个候选，已回滚所有 SKILL 镜像改动。

## 结论

本轮不产生发布候选。当前证据显示，单纯压缩或拆分 `SKILL.md` 事实边界段不是可靠方向：压少了不改善，压多了会撞回归护栏或削弱事实边界。

后续建议：

- 先不要继续改 `SKILL.md` 入口事实边界。
- 若继续做压缩，应优先处理按需 reference 的结构可读性，例如给 `workflow.md:95` 拆成小标题和短段，但不改变语义；这种改动也必须跑弱/强模型真实 A/B。
- 对弱模型首稿标准可按用户要求放宽到“第一轮基本可用，第二轮指出问题后可修”，下一步更值得测试二次修改链路，而不是继续堆入口 prompt。
- 如果要借鉴社区方案，优先借鉴 `plan-proofread` 的“入口索引 + reference 细则 + 审稿清单化”，不借鉴重脚本、硬评分和硬禁词。
