# 2026-07-09 深渐进式披露与微型路由卡 A/B 测试记录

## 目的

在多轮 `SKILL.md` 入口压缩、长句拆分和 `workflow.md` reference 拆段均回退后，本轮不再修改产品 prompt，先测试两种路径：

1. **加深加载路径**：同一组真实 prompt 下，分别控制是否读取整份 `workflow.md` 和 `genre-playbooks.md`，观察“多读 reference”是否能提升弱模型稳定性。
2. **细颗粒路由卡**：不读整份 reference，只给当前任务需要的微型规则卡，观察更细颗粒的按需披露是否减少弱模型注意力负担。

这两者不是一回事。用户所说的“深渐进式披露”更接近第二种：把 reference 按需加载策略做得更细，而不是让弱模型加载更多无关内容。

## 阶段一：加深加载路径

- A：只读 `SKILL.md` 入口。
- B：`SKILL.md` + `references/workflow.md`。
- C：`SKILL.md` + `references/workflow.md` + `references/genre-playbooks.md`。

弱模型使用 `gpt-5.3-codex-spark low`，强模型使用 `gpt-5.5 low`。所有 writer 只读成稿，不修改仓库。

## Prompt 覆盖

### 任务 1：材料稀疏型情况说明

写《版慎通数据治理专项推进情况说明》。材料只有：

- 已接入 3 个业务系统。
- 发现重复字段 120 项。
- 目前只完成初步梳理。
- 尚未形成整改结论。
- 没有责任部门、时间表、验收标准。

要求正文完整，但不要编造工作组、问题清单、统一共识、治理流程、验收节点、责任单位或完成时限。

### 任务 2：未决事项会议纪要

整理会议纪要。材料：

- 主题：版慎通模型成本治理。
- 时间：2026 年 7 月 9 日上午。
- 地点：第一会议室。
- 参会：技术应用部、财务部。
- 内容：当前一个月模型成本约 6 万元，贵价模型占比较高，拟先评估低思考档位模型和分级模型机制；下次会议听取测试结果。

要求不要写“会议强调”“会议认为”，不要补责任分工、完成期限、采购方案或考核要求。

## Verifier 结论

独立 verifier 只看原 prompt 和 6 组输出，判定如下：

| 路径 | 弱模型 | 强模型 |
| --- | --- | --- |
| A：入口 | WARN | WARN |
| B：入口 + workflow | FAIL | WARN |
| C：入口 + workflow + playbook | FAIL | PASS |

### 主要问题

- weak A：情况说明尾部接近补“相关业务、推进计划”等执行安排；会议纪要把“拟先评估”写成“决定先评估”。
- weak B：写出“已形成问题清单”，违背“不要编造问题清单”；又要求“明确后续整改路径”，并把审批链、责任部门等列进补充项。
- weak C：出现 `###`、`**`、`---` Markdown 残留；补出“统一入口”“字段重复定义与边界重叠”“责任、优先级和办理路径推进”等未给事实。
- strong A：整体可用，但补了“字段合并、口径统一、重复字段来源、使用场景、影响范围”等轻度治理方向。
- strong B：整体可用，但会议纪要写“认为有必要”，触碰用户禁止“会议认为”的要求。
- strong C：整体最稳，把整改、验收、处置都放在“尚需核实、以正式审定为准”的边界内，会议纪要也未补责任、期限、采购或考核。

## 判断

不存在“多读 reference 单调改善”。弱模型从 A 到 B/C 明显恶化，B 增加事实外扩，C 增加 Markdown 残留；强模型 C 最好，但 B 中间仍有禁止项触碰。

本轮不建议修改 skill 去强制或鼓励加载整份 `workflow.md`、整份 `genre-playbooks.md`。对弱模型而言，增加整份 reference 没有改善，反而可能加重注意力负担和材料外扩。对强模型而言，深加载有局部收益，但不足以覆盖弱模型退化风险。

## 阶段二：细颗粒微型路由卡

用户进一步澄清：深渐进式披露不是“读更多 reference”，而是把 reference 按需加载策略做得更细，避免加载当前写作无关内容。按这一口径，本轮追加了一次最后的弱模型一次成稿测试。

微型路由卡只包含当前 3 个任务需要的 5 条短规则：

1. 通用事实边界：先完成正文，缺失责任单位、完成期限、流程、验收、结论、联系人、反馈渠道时不写进正文。
2. 稀疏情况说明：只说明已给事实和“尚未形成结论”，不补问题清单、工作组、治理流程、整改路径、执行安排。
3. 会议纪要：只记录主题、时间、地点、参会、会议内容和下次安排；“拟、建议、听取测试结果”保持未决口径。
4. 限字通知：保留对象、时间、渠道、联系人、电话、落款、日期；不补“认真落实、严肃处理、记录留痕、无论有无异常”等执行链条。
5. 格式：正式正文不用 Markdown 加粗、`###`、`---` 和代码块。

### Prompt 覆盖

在阶段一两个 prompt 基础上，增加第 3 个限字通知：

把内部账号安全清查材料压成 220 字以内通知，保留 OA、联系人、电话、落款和日期，不补“认真落实、严肃处理、记录留痕、无论有无异常”等执行链条。

### 结果

独立 verifier 按用户更新口径判定：弱模型重点看内容是否符合 prompt、事实边界和禁止项；Markdown 残留、标题包装、横线等格式噪点只记 WARN，不作为弱模型一次成稿 FAIL 主因，除非影响正式内容理解。

| 路径 | 判定 | 说明 |
| --- | --- | --- |
| weak micro | PASS | 三项均贴住 prompt 核心约束。任务 1 未补工作组、问题清单、治理流程、责任单位、时限、验收节点；任务 2 保留“拟评估”口径；任务 3 保留 OA、联系人、电话、落款日期，未补执行链条。 |
| strong micro | WARN | 任务 2、任务 3 基本可用；任务 1 的“需核对重复字段来源、使用场景和影响范围”属于轻度主动展开，未直接补责任部门或完成期限，但仍有滑向后续治理安排的风险。 |

## 综合结论

## 后续路线

- 不再把入口压缩、reference 拆段或整份 reference 深加载作为默认修复方向。
- 细颗粒微型路由卡值得作为后续设计方向，但本轮只记录证据，不直接把微型卡塞进正式 skill，避免未经更大消融就引入新结构。
- 弱模型一次成稿不再死磕格式噪点；重点看内容是否符合 prompt、事实边界和禁止项。Markdown 残留这类无关内容质量问题可按 WARN 记录。
- 弱模型整体交付口径应转向“首稿基本可用，二次/三次自然语言局部修改后可交付”，而不是追求一次成稿达到强模型水平。
- 后续若继续测试，应固定加载路径或微型卡片，分别用弱/强模型做同题 A/B，重点观察：
  - 材料稀疏型说明/通报不补处置链条。
  - 会议纪要严格保留“拟、建议、听取测试结果”等未决口径。
  - 限字通知保留渠道、联系人、电话、落款、日期。
- 如果要修改 prompt，每次只改一个小块；如果三次修改仍回退，应停止 prompt 堆叠，继续二次/三次修改链路或只读 final-draft inspection 路线。
