# URL 来源摘录与稀疏报告诊断（2026-07-08）

## 目的

上一轮 `delivery-fact-check` 候选已回退。为避免继续堆 prompt，本轮只做诊断测试：区分问题来自“URL 来源没有稳定读取”，还是来自“稀疏事实组织时自动补链条”。

## 方法

使用 `gpt-5.4-mini`、低思考、隔离 writer subagent。writer 加载当前 `chinese-official-writing/SKILL.md`，不修改仓库。

本轮不再只给 URL，而是把两个网页的关键事实摘录直接放入 prompt：

- Claude Code 风险提示：安全后门、回传地域和身份标识、受影响版本 `2.1.91 至 2.1.196`、卸载或升级安全版本、7 月 10 日 15:00 钉钉报技术应用部。
- 暴雨橙色预警：7 月 8 日 06 时预警、7 月 8 日 08 时至 9 日 08 时、多个地区大到暴雨、广西东南部沿海局地特大暴雨、短时强降水和雷暴大风。
- 对照样本仍保留 OpenClaw 稀疏报告，只给“已排查，未发现安装 OpenClaw 及任何小龙虾变种软件，报江西出版集团，发送方红星出版社”。

独立 verifier 只看 prompt 和成稿，不看主上下文判断。

## 结果

### A. Claude Code 来源摘录改通知

判定：WARN。

版本号、风险、报送时间、渠道、报送对象、落款均保留。仍有轻微外扩，例如 `工作终端及相关设备`、`同步核实其他安全版本`、`不漏一台设备` 等。

### B. 气象来源摘录改通勤提醒

判定：PASS。

预警时间、预报时段、地区范围和主要天气风险基本保留；删去小时雨量等细节属于通知场景下可接受压缩。

### C. OpenClaw 稀疏报告

判定：WARN。

文本较短，但仍补入 `按照有关要求`、`相关终端设备` 和当前日期。说明稀疏报告、情况说明或已排查类材料仍容易自动补背景依据、排查范围和正式要素。

## 结论

1. URL-only 测试不应同时评估网页读取能力和公文写作能力。若测试目标是公文写作，应给来源摘录或确认 writer 具备联网读取能力；否则失败可能来自未读取来源，而不是 skill 文种能力。
2. 提供来源摘录后，URL 改写质量明显改善，说明继续往 `SKILL.md` 堆 URL 事实抽取规则的边际收益很低。
3. 真正仍需处理的是稀疏事实下的报告类组织：模型会补依据、范围、日期、对象载体、后续管理动作。
4. 下一轮不建议继续新增“场景名称 + 禁止项”的首稿 prompt。更合适的方向是：
   - 把 URL 回归测试拆成“来源读取/摘录”和“公文改写”两个环节；
   - 聚焦稀疏报告的二次修订或外部 final-draft inspection；
   - 如果修首稿，只能做更小、更通用的事实映射机制，先在弱模型真实样本上验证，失败即回退。
