# 结构化二次修订交互测试记录（2026-07-08）

## 目的

连续多次 prompt/reference 修补回退后，参考社区技能中的“诊断 -> 修改 -> 完整改后版本”“逐条修改”“修改阶段和格式阶段拆开”等思路，不先改产品文件，先测试一种更结构化的二次修订交互是否能让弱模型达到“用户指出问题后二次修改可用”。

## 测试形态

要求弱模型对 C1/C2 问题稿先输出三行短单：

- `保留：` 用户给定事实和动作。
- `删除：` 未授权字段、流程、格式痕迹。
- `归位：` 标题、主送或报告对象、正文、落款、日期。

随后输出 `完整改后版本`。

## 测试结果

### C1 Claude Code 排查通知

用户要求：

- 修干净、能发，去掉格式痕迹和没说的内容。
- 如有请删除，未发现也报无。
- `7月10日15:00通过钉钉发送至技术应用部`，不要写 `15:00前`。
- 无法联网核验版本清单时，只写相关版本以原链接所列范围为准。

弱模型输出问题：

- 短单声称删除推断性内容，但完整稿仍保留 `主机名称、用户、安装路径`。
- 仍保留 `报送内容至少包含：是否发现、发现版本、数量、是否完成处置、未发现原因说明`。
- 把 `7月10日15:00通过钉钉发送至技术应用部` 改成 `2026年7月10日通过钉钉统一报送至技术应用部`，漏掉 `15:00`。

独立 verifier：`FAIL 倾向`。

### C2 OpenClaw 排查报告

用户要求：

- 只保留已排查、未发现本单位安装有 OpenClaw 及其任何小龙虾变种软件。
- 报告对象江西出版集团，发送方红星出版社。
- 不补核查范围、安装记录、残留文件、自启项、进程、后续巡检或月度复核。
- 主送、正文、落款和日期放在可发文位置。

弱模型输出问题：

- 删除了多数禁补项，核心结论保留。
- 仍保留 `报 告 人（发送方）`、`报 告 对 象` 字段痕迹，没有把 `江西出版集团：` 放到自然主送位置。

独立 verifier：`WARN`。

## 结论

结构化二次修订交互有启发，但没有达到发布级“弱模型二次修改可用”口径。核心失败点不是不知道要删，而是短单和最终稿不一致：删除项仍进入完整稿，用户点名时间被改漏。

本轮不写入 skill，不新增 prompt/reference 规则，不 bump 版本号。

## 下一步建议

- 不要继续在弱模型上叠加“更多提示形态”来解决同一问题。
- 若必须兜底弱模型交付质量，下一轮应比较外部 cleaner、强模型二修或工具化差异检查，而不是继续修改默认 skill。
- 如果未来引入结构化二修，只能在真实测试证明可用后最小加入；且必须保证短单中的 `删除` 项不得出现在完整改后版本里。
