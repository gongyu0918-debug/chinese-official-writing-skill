# 二次修改路径对比记录（2026-07-09）

## 背景

连续多轮把规则写入 `SKILL.md` 或 `review-checklist.md` 后，确定性消融可通过，但弱模型真实二修仍会保留字段痕迹、Markdown 残留和未授权报送字段。本轮不修改产品文件，只对比三条路径：

1. 弱模型继续按当前 skill 自行二修。
2. 强模型直接按用户二次修改后的要求重写最终稿。
3. 外部删减型 cleaner 只看原始需求、二次修改指令和弱模型稿件，负责删除、归位和清理，不新增内容。

测试覆盖两个上一轮反复失败的真实场景：

- Claude Code URL 改本单位通知：要求排查受影响版本、如有请删除、没有也汇报无、`7月10日15:00` 通过钉钉发送至技术应用部。
- OpenClaw 排查报告：仅给出未发现安装结论、报告对象和发送方，要求不补排查流程、范围、责任人或后续机制。

| OpenClaw 报告 cleaner 稿 | PASS | 改成正常报告格式；无字段标签；未补流程、范围、责任人或后续机制；短、稳、可上报。 |

保留观察：cleaner 仍把 `7月10日15:00` 写成 `7月10日15:00前`。verifier 本轮判定为可接受，但用户曾多次强调定点时间不要变形；后续如果继续测试 cleaner，应把“定点时间不改成截止时间”列为单独观察项。

## 本轮结论

1. 不修改 skill。本轮只说明强模型二修和外部删减型 cleaner 值得继续压测，样本数量不足以支撑写入默认 workflow。
2. 弱模型“自行二修”仍不稳定，不能作为发布口径。
3. 外部 cleaner 相比弱模型二稿更稳，但仍需更多稀疏材料、URL 改通知、报告、赛事提醒和普通说明场景验证，尤其要看是否会改动用户指定时间、发送方式和事实口径。
4. 如果下一轮继续，应只测试“用户明确要求修干净、删掉没说的内容、只输出能发正文”这一二修场景，不把 cleaner 变成首稿默认阶段，不写入 `SKILL.md` 入口。

## 扩展样本：情况说明、运营情况说明、赛事提醒通知

为避免只在 C1/C2 两个历史失败样本上得出结论，继续增加 3 个真实用户式场景：

- 电脑采购情况说明：用户给出采购 5090 的理由，并要求不要写成请示、不补采购流程和验收安排。
- 微博情况说明：用户给出数据、阅读高峰、本地新闻流量弱和下一步预计，要求不补运营机制和考核指标。
- 世界杯期间通知：用户只给禁止赌球、强化考勤和祝观赛愉快，要求语气柔和，不补巡查、上报、反馈、值班或负责人责任。

独立 verifier 判定如下：

| 场景 | 弱模型自行二修 | 强模型直接二修 | 外部 cleaner |
| --- | --- | --- | --- |
| 电脑采购情况说明 | WARN | PASS | PASS |
| 微博情况说明 | WARN | PASS | PASS |
| 世界杯期间通知 | WARN | WARN | PASS |

主要观察：

- 弱模型二稿能保留核心要点，但连续出现 Markdown 行尾空格、标题语感不稳、格式不够正式或语气不够柔和的问题。
- 强模型直接二修在情况说明类稳定，但通知类仍可能顺手补轻微部门动作，例如“提醒员工合理安排观赛时间”，且正式落款日期不稳。
- 外部 cleaner 在 3 个扩展样本中全部 `PASS`，尤其能删外扩、压短、去 Markdown，并较好控制通知类不补巡查、处理、上报、反馈、值班和负责人责任。
- cleaner 仍不是可立即写入 skill 的证据。它更像一个用户触发的二修路径候选；若要进入产品规则，下一步还要继续压测定点时间、落款日期、主送归位和是否误删必要事实。

更新后的路径判断：

1. 外部 cleaner 是目前最稳定的候选方向，5 个样本中 verifier 均给 `PASS`。
2. 强模型直接二修适合复杂返工，但在通知类管理动作上仍有轻微外扩风险。
3. 弱模型自行二修可作为“首稿基本可用”或“初步二修”，但不应作为最终发布质量口径。
4. 仍不修改 skill。当前证据足以支持下一轮继续压测 cleaner/强模型二修路径，不足以把它写进默认工作流。

## 更苛刻样本：定点时间、禁止日期、来源摘要

继续只测试外部 cleaner，不修改产品文件。新增 3 个薄弱点样本：

- TestAgent 通知：用户强调 `7月12日15:30` 是发送时间点，不是 `15:30前`；要求不能补主机名、用户、路径、附件表、负责人或反馈模板。
- LanternAI 报告：用户明确不要成文日期，只给“未发现安装”这一事实，要求不补排查范围、过程、责任人或后续机制。
- NoteCoder 来源摘要通知：要求保留 `1.4.0、1.4.1、1.4.2` 三个版本号，保留来源摘要但不要写成已核验权威结论，不补反馈字段、统计口径、责任部门或后续整改。

弱模型二稿先暴露了这些问题：

- D 为避免 `15:30前`，额外写出“请不要在截止时间前提前汇报，需在该时间点提交”，语义不自然。
- E 丢失正常主送格式，写成“蓝星出版社报告”。
- F 保留版本和来源摘要，但落款缺日期。

cleaner 清稿后，独立 verifier 判定如下：

| 场景 | cleaner 结果 | 主要理由 |
| --- | --- | --- |
| D 定点时间通知 | PASS | 保留 `7月12日15:30通过钉钉`，未改成截止时间；未补主机名、用户、路径、附件表、负责人、反馈模板；落款日期正确。 |
| E 禁止日期报告 | PASS | 正常报告格式；未补成文日期；未出现字段标签；未补排查范围、过程、责任人或后续机制。 |
| F 来源摘要通知 | WARN | 版本号完整，未补反馈字段和责任机制，来源仍写成摘要；但“根据来源摘要”略口语，不够正式。 |

阶段性判断：

1. cleaner 在更苛刻样本中仍能明显压住“时间变形、乱补字段、乱补日期、乱补机制”风险。
2. 残留风险从事实外扩转为表达正式度，主要是来源摘要句式不稳。
3. 当前累计 8 个 cleaner 样本：7 个 `PASS`、1 个 `WARN`、0 个 `FAIL`。这足以继续推进 cleaner 路径压测，但仍不建议直接写入默认 skill。
4. 下一轮如要进入修改阶段，应考虑“用户明确要求修干净/只输出可发正文时，可触发最终 plain-text cleaner”这种窄路径，而不是继续在普通起草流程中堆 prompt。
