# 当前可用程度测试：6 个真实 prompt（4f8f4ed）

日期：2026-07-08

目的：在当前本地基线 `4f8f4ed` 上，用用户指定的 6 个真实写作 prompt 观察实际可用程度，重点看指令遵循、要点置入、事实边界、格式残留、URL/联网场景和弱模型稳定性。本轮只测试并记录问题，不修改 skill 规则。

## 方法

- writer A：`gpt-5.3-codex-spark`，低思考，加载 `chinese-official-writing` skill，分别生成 6 篇短稿。
- writer B：`gpt-5.5`，低思考，加载 `chinese-official-writing` skill，分别生成 6 篇短稿。
- verifier：`gpt-5.5`，低思考，只看“原 prompt + 两组输出”，不重写稿件，按 PASS/WARN/FAIL 判定。

6 个 prompt 覆盖：

1. 5090 电脑采购情况说明。
2. 微博运营情况说明。
3. URL 改写为 Claude Code 版本排查通知。
4. URL 改写为上下班天气防范通知。
5. 世界杯期间禁止赌球、强化考勤但祝观赛愉快的通知。
6. OpenClaw 安装情况排查报告。

## Verifier 结论

总体：当前 skill 对强模型可用，对弱模型不宜作为稳定发布口径。强模型 6 个场景整体能按文种成稿，事实边界基本可控；弱模型在事实外扩、流程机制补造和格式克制上仍不稳定，尤其是 P3、P4、P6 这类“由素材改通知/报告”的场景。

| 场景 | 弱模型 | 强模型 | 主要判断 |
| --- | --- | --- | --- |
| P1 电脑采购说明 | WARN | PASS | 弱模型要点基本齐，但题名扩大为“服务器端扩展用计算节点”，且补落款单位；强模型准确保留采购理由、用途和成本安全逻辑，未明显外扩。 |
| P2 微博情况说明 | WARN | PASS | 弱模型保留数据，但补“早间预告+晚间收官”、落款单位，属未给运营机制和主体；强模型数据、趋势、下一步都较稳。 |
| P3 Claude Code 排查通知 | WARN | WARN | 弱模型补“截图留存、版本登记、隔离更新、责任人、排查范围”等未给流程；强模型整体可用，但补“排查设备数量、联系人”和泛称“软件安全风险提示”，且未基于 URL 具体版本信息展开。 |
| P4 气象文章改通知 | WARN | PASS | 弱模型补“微信群、钉钉群、邮件同步发布并督促执行”，未给发布机制；强模型没有编造原文来源，改成上下班防范通知较自然。 |
| P5 世界杯期间通知 | WARN | PASS | 弱模型补“必要观赛活动、值班替代、周例会比赛间隙安排”，越过用户要求；强模型能同时覆盖禁赌球、考勤和祝观赛愉快，口径稳。 |
| P6 OpenClaw 排查报告 | FAIL | WARN | 弱模型补排查范围、核查方式、风险评估、季度例检机制，事实外扩严重；强模型也补了排查范围、后续审批和安全排查机制，报告可读但事实边界不够稳。 |

## 共性风险

1. 稀疏材料被补成完整管理链条：排查、处置、报送类材料里容易补“范围、责任人、截图留存、登记、审批、例检、后续机制”等未给事实。
2. 正式文种完整性压过事实边界：模型为了像正式文件，会补落款单位、日期、部门名称、联系人、报送字段、发布渠道等，弱模型更明显。
3. URL 改写场景没有真正落实来源边界：两组输出都没有基于 URL 中具体内容或版本范围作可靠承接，强模型用“相关版本”“安全风险提示”等泛称回避，弱模型则进一步补处置流程。对 URL 场景，当前只能判定为“可生成通知框架”，不能判定为“已准确改写来源内容”。

## 当前判断

- 强模型：整体 `PASS with WARN`，可作为辅助起草使用，但 P3/P6 仍需人工复核事实边界。
- 弱模型：整体 `WARN`，P6 为局部 `FAIL`，不建议作为发布级稳定能力口径。
- 发布判断：本轮只记录当前可用程度，不建议发布，也不建议继续往 `SKILL.md` 追加长规则。后续若修复，应围绕“稀疏排查/处置/报送材料不补管理链条”和“URL 指代必须核验具体范围，否则正文外提示”做最小修改，并用同 6 个 prompt 复测。
