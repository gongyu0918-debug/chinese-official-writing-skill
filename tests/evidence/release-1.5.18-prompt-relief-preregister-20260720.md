## 工程门

产品提交后先运行：

1. 入口字符统计和重复规则仍在 reference 的静态核验；
2. `python -m unittest discover -s tests`；
3. `npm run eval:official-writing:smoke`；
4. 固定 1.5.17 确定性消融；
5. `tools/sync_adapters.py` 后 canonical/镜像一致性；
6. `quick_validate.py chinese-official-writing`；
7. `git diff --check`。

工程门失败时停止，不进入真实写稿。

## 真实 A/B

复用 R18-01 报告、R18-02 纪要、R18-03 通知三个自然任务。Candidate 与固定 1.5.17 均使用 `gpt-5.6-terra`、medium、逐字一致原始任务和同构宿主委托；两侧只替换 Skill 根目录，不在 Candidate 委托中额外写门禁机制、目标答案或风险词。各取首个技术有效输出，不补抽。

独立盲审只接收原始任务和随机匿名稿，先核验事实、数字、日期、主体、状态、文种、格式、篇幅、输出模式、标记残留、空稿和循环，再比较结构、可读性、重复解释、保护性外扩和直接修改成本。

## 验收门

同时满足以下条件才允许把“Hook + 入口减负”作为 1.5.18 发布候选：

1. 三题 Candidate 技术有效，事务有界、终稿非空、无标记残留；
2. 三题均无 Candidate 独有的事实、数字、主体、状态、文种、格式、篇幅或输出模式硬回退；
3. Candidate 至少 2/3 明确优于 1.5.17，且没有任何一题整体直接使用成本更高；
4. 至少一题产生被语义层接受且相对 D0 有净收益的 D1；
5. 相对 Hook 改前入口，字符减少达到 5%—10%，确定性能力和镜像一致性无回退。

任一硬门失败即撤回减负产品提交；若最终组合不能优于 1.5.17，则阻断 1.5.18，不升版、不打 tag、不发布。
