# 交付修订模式最小修复循环记录（2026-07-08）

## 背景

本轮目标是在不增加硬阻断、不膨胀默认写稿流程的前提下，测试弱模型低思考下的真实写稿能力，并选择性保留最小修复。

## 首稿真实写作测试

使用 3 个弱模型 writer subagent，覆盖 6 个真实 prompt：

1. RTX 5090 电脑采购情况说明。
2. 微博运营情况说明。
3. Claude Code 受影响版本排查通知。
4. 暴雨预警改上下班路上防范通知。
5. 世界杯期间禁止赌球、强化考勤但祝观赛愉快通知。
6. OpenClaw 安装排查报告。

独立 verifier 判定：2 个 FAIL、4 个 WARN、0 个可直接无修改交付。三次以上共性问题为：

- Markdown/格式残留。
- 稀疏材料补造排查范围、执行链条、制度化措施。
- 通知/报告类容易把建议或提醒扩成已定安排。

结论：弱模型首稿仍只能作为初稿，不能作为发布充分证据。

## 回退的候选

曾尝试在 `SKILL.md` 顶部加入“交付前三项短检”（格式净稿、事实净稿、文种和主体复核），并加入 P097 确定性守卫。确定性消融通过，但真实复测仍出现事实外扩和格式残留，独立 verifier 判定不通过。

处理：已回退该默认首稿短检和 P097，不纳入候选。

## 保留的候选

保留 `references/review-checklist.md` 的“交付修订模式”，仅在用户把已有草稿发回并要求“修干净、能发、删掉没说的东西、去掉格式痕迹、按已给材料重改”时触发。该模式明确：

- 二次修订动作，不作为默认成稿前阶段。
- 删除 Markdown、代码块、横线、行尾双空格和过程痕迹。
- 删除草稿中未给的排查范围、检查方法、值班巡查、备案、长期复核、反馈字段、处分口径、原因结论。
- 用户说“只保留”时，只保留其点名事项，不补制度依据、设备账号、网络渠道、负责人、值守交接、处置程序或来源化表述。

## 二次修订真实测试

使用弱模型 writer subagent 对 4 个坏稿做二次修订，再由独立 verifier 判断：

- Claude Code 通知：WARN。大部分补造链条被删，但仍有个别行尾空格和不自然表述。
- OpenClaw 报告：PASS。删除未给检查范围和治理流程，保留核心结论。
- 世界杯通知：WARN。删除主要强执行链条，但仍有少量制度化表达。
- 暴雨防范通知：PASS。标题、来源和落款修正，删除未给安排。

后续对 A/C 两个 WARN 复测，仍观察到行尾双空格偶发和落款位置不稳。结论：交付修订模式有明显二次修复价值，但不能宣称完全解决弱模型首稿问题。
