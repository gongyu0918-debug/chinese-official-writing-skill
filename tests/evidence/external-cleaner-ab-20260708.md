# 外部 cleaner 交付前清稿 A/B 测试记录

日期：2026-07-08

起点：`03eadbb test: record failed second-revision chain`

本轮目标：不继续修改 `SKILL.md` 或 reference，测试“外部 cleaner/交付前清稿”是否比让弱模型 writer 自行二次修改更稳定。

## 测试方法

待清稿材料来自上一轮弱模型二次修改失败样稿，覆盖 4 类问题：

1. Claude Code 通知：Markdown 残留、定点时间被改成截止时间、来源说明混入正文。
2. 气象通知：删除居家办公、户外作业、值班联系人和来源说明。
3. 世界杯通知：补回禁止赌球，删除日报、值班岗位等新增要求，保留祝观赛愉快。
4. OpenClaw 报告：只保留已给排查结论、报告对象和发送方，不写全量终端、服务项、巡检清单等过程。

cleaner 提示使用极短约束：

- 只做清稿，不重写新内容，不解释过程，不输出问题清单。
- 删除 Markdown 标记、括号式说明、来源说明、过程说明。
- 删除用户没给的管理动作和排查过程。
- 把被改坏的精确时间、动作、落款改回用户原话。
- 直接输出清稿后的正文。

writer A：`gpt-5.3-codex-spark`，low reasoning，加载当前 skill。

writer B：`gpt-5.5`，low reasoning，加载当前 skill。

verifier：独立上下文 `gpt-5.5`，low reasoning，只看原始要求和两个 cleaner 输出。

## 结果

独立 verifier 结论：总体 WARN。

| cleaner | 结论 | 主要证据 |
| --- | --- | --- |
| 弱 cleaner | WARN/FAIL 边界 | 仍保留 `**通知**`；任务 1 把 `7 月 10 日 15:00 发送` 改成 `15:00 前`；对精确时间和正文洁净度仍不稳。 |
| 强 cleaner | PASS | 清除了 Markdown、括号说明和来源说明；保留版本区间、删除/无也汇报、钉钉、技术应用部、落款日期；气象和 OpenClaw 场景未引入禁写项。 |

## 处理决定

外部 cleaner/交付前清稿流程值得作为后续候选继续验证，但暂不写入 skill 默认流程。

原因：

- 弱 cleaner 对最基础的 Markdown 清除和精确时间还原仍不稳定。
- 强 cleaner 效果明显更好，说明流程方向有价值，但它更像发布前实验流程或人工复核辅助，不应直接变成所有写作任务默认步骤。
- 若把 cleaner 写进 skill 默认 workflow，可能造成流程变重、弱模型注意力偏移，或让模型在正文后追加解释。

下一轮建议：

- 只在发布前真实评测中保留“强 cleaner + 独立 verifier”作为候选流程。
- 若要进入 skill，也应先做更大样本 A/B，证明它不会导致普通起草、只审不改、二次修改和长文限字场景回退。
- 在有充分证据前，不新增 prompt 规则、不新增 P 用例、不发布。
