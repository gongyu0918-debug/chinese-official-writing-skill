# 一致性检查相关 skill 市场检索记录

日期：2026-07-07

本轮目标：为中文公文写作 skill 的“稿内一致性、事实边界、二次修改链路”寻找可最小借鉴的社区 skill 思路。只借鉴检查维度、报告结构和 prompt 组织方式，不复制代码、脚本、正则、模板库或硬门禁。

## 检索范围

- ClawHub CLI：`consistency`、`proofread`、`fact check`
- AI Skill Market：公开目录、ClawHub Top 500、llms.txt
- MCP Market：Agent Skills 目录
- GitHub：`document consistency checker skill`、`SKILL.md proofreader`、`文档一致性 校对`

## 拒绝项

- 不引入自动 docx 批注、一键修正、版本备份、标准库入库或脚本流水线。
- 不引入 P0/P1/P2 作为发布硬门禁；本技能继续用“硬边界/质量建议/待确认”三层。
- 不把事实核查 skill 的默认搜索流程嵌入写稿；联网仍只用于用户明确要求、时效事实或来源核验。
- 不采用小说角色一致性、医学语义相似度、论文引用验真等领域专用模板。

## 本轮最小借鉴方向

1. 增加“稿内一致性轻量复核”软规则：主体、事项、金额/数据、时间、责任、附件、术语、章节提示是否前后一致。
2. 增加“事实映射式二次修改”软规则：二次修改时把每个实质句分成“用户已给事实 / 直接概括 / 未支持推断”，未支持推断删除或移到正文外待确认。
3. 对材料稀疏型通报/情况说明/报告，优先提供可直接仿写的短正文形态，而不是继续增加禁止词清单。

## 下一步验证

- 先用弱模型测试二次修改链路是否能删除未给事实。
- 若二次修改有效，将风险处理下放到“事实对齐二次修改”而不是继续加重起草规则。
- 若二次修改仍无效，再考虑更明确的短正文示例，但仍保持为 prompt 软约束，不新增硬阻断。
