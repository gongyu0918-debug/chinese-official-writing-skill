# 二次修改链路真实测试失败记录

日期：2026-07-08

起点：`22c0ac8 test: record failed weak-model prompt loop`

本轮目标：不继续叠加入口 prompt 规则，改测现有“用户指出补造事实后进行二次修改”的链路，看弱模型能否在自然语言返工场景中删掉未给事实、保留用户要点、只输出干净正文。

## 测试方法

writer：`gpt-5.3-codex-spark`，low reasoning，加载当前 `chinese-official-writing` skill。

verifier：独立上下文 `gpt-5.5`，low reasoning，只看用户返工请求和弱模型输出。

测试不修改仓库，不使用专业术语提示弱模型，只用真实用户可能会说的话。

## 4 个返工任务

1. Claude Code 通知二次修改：用户要求删掉“删除或升级”，只写发现即删除；不要把 `7月10日15:00发送` 改成 `15:00前`；不要来源说明和指定专人复核。
2. 气象通知二次修改：只提醒上下班路上注意防范，删除居家办公、户外作业、值班联系人和来源说明，落款正常放最后。
3. 世界杯通知二次修改：补回“禁止赌球”，删除日报、值班岗位等新增要求，语气柔和，保留祝观赛愉快。
4. OpenClaw 报告二次修改：只保留已排查、未发现安装 OpenClaw 及任何小龙虾变种软件，保留报告对象和发送方，不写全量终端、服务项、巡检清单等过程。

- 任务 1：仍保留 `**通知**` Markdown 加粗和“（正文应直接发，不保留来源说明）”括号说明；仍把 `7月10日15:00发送` 改成 `7月10日15:00前通过钉钉报送`。

- 任务 3：补上禁止赌球，但仍写“强化考勤管理”“确保正常值班与到岗”；落款从红星出版社办公室变成红星出版社。
- 任务 4：基本短写，只保留报告对象、发送方和未发现结论。

## 独立 verifier 结论

总体：FAIL。

| 任务 | 结论 | 主要原因 |
| --- | --- | --- |
| Claude Code 通知 | FAIL | 残留 Markdown 加粗和括号说明；定点时间改成截止口径；仍不够干净。 |
| 气象通知 | PASS | 基本删除未给安排和来源说明。 |
| 世界杯通知 | FAIL | 仍保留或换写值班/考勤新增安排，语气不够柔和，落款走样。 |
| OpenClaw 报告 | PASS | 基本只保留已给结论、报告对象和发送方。 |

两次以上共性问题：

1. 对用户明确要求删除或不要写的内容，仍会保留或换一种说法写回正文。
2. 二次修改中会改写用户给定的精确信息或补入未给要素，例如定点时间改成截止时间。
3. 正式正文洁净度不稳，仍可能残留 Markdown 加粗、括号式说明或过程提示。

## 处理决定

本轮不做 prompt 修改，也不新增 P 用例。原因是失败点更像弱模型二次修改指令遵循不稳定，而不是单条规则缺失；继续堆 prompt 容易扩大对强模型和正常场景的干扰。

下一步建议先把发布判断降级为：当前版本适合强模型或人工复核辅助场景，不适合宣称弱模型低思考下稳定可发布。若继续改，应优先探索“交付前只输出正文的极短修订提示”或外部 verifier/cleaner 流程，而不是继续膨胀 `SKILL.md`。
