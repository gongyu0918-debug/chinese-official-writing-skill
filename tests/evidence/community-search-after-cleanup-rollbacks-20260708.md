## 可借鉴点

### 1. 诊断先于改写，但诊断要短

来源：

- `writing-polish`：工作流程为通读理解、诊断、动手修改、交付；诊断只说最重要的 2-3 个问题。
- `govwriting`：AI 润色工作流先做初稿诊断，并按 P0/P1/P2 排列。

适配判断：

- 可借鉴“二次修改先做短诊断，再改正文”的形态。
- 不应把诊断作为默认首稿阻断，也不输出长表格。
- 对当前问题，弱模型清稿失败不是因为不知道规则，而是没有形成“先定位待删项再改”的动作。短诊断可能比继续加规则更有效。

### 2. 逐条修改用户意见，而不是再生成一遍

来源：

- `govwriting` 的审核定稿阶段强调逐条修改、再次通读。
- `improvewriting` 的输出可用“修改对照”和“完整改后版本”。

适配判断：

- 当前二修失败集中在用户明确说“删掉安装路径、报送字段”等，但模型仍保留。下一轮可测试“先列用户意见落实清单，再给完整改后版本”。
- 这应只在用户要求返工、修干净、指出具体问题时触发；不能影响正常首稿。
- 最小实现不应新增脚本或强制审稿表，只在真实测试 prompt 中验证形态是否有效。

### 3. 修改阶段和格式阶段拆开

来源：

- `govwriting` 的流程清单把写作、修改润色、格式规范、审核定稿分阶段。
- `writing-polish` 明确格式克制，少用 Markdown、分隔线和过深小标题。

适配判断：

- 当前失败说明“内容删减”和“格式归位”混在一个 prompt 里弱模型容易漏项。
- 下一轮可测试二步式二修：第一步只删未授权字段，第二步只做标题/主送/正文/落款/日期归位。
- 不建议写成 skill 默认硬阶段；只作为二次返工的交互策略或测试路径。

### 4. “无需修改”机制可防止过度清洗

来源：

- `improvewriting` 有 no issues found 路径。
- `writing-polish` 提到原文已经很好时不要强行修改。

适配判断：

- 对公文技能有价值：避免为了执行清稿而删掉用户真实模板、字段式材料或正式落款。
- 可作为 verifier 口径：如果用户模板本身可用，二修只改被指出的问题。
- 不应引入“评分”或硬门槛。

### 5. 事实弱时降级判断，不补执行链条

来源：

- `official-document-skill` 明确弱证据时降低判断强度，并要求先抽取事实。

适配判断：

- 本技能已有类似规则，但弱模型仍补执行链条。下一轮不应重复写同义规则。
- 可借鉴“先抽取事实再改”的外显过程，在二修时要求先列“保留事实/删除事实外扩/格式归位”三行短单，再输出正文。

## 拒绝项

- 拒绝 `official-document-skill` 的内部评分和低于 80 分重写机制：容易变成伪精确和内部自检循环。
- 拒绝 `govwriting` 的大规模模板、金句、词组库和格式手册整体搬运：会使本技能膨胀，破坏渐进式披露。
- 拒绝 `writing-polish` 的口语化、人味化和个人特色保留规则直接迁移到正式公文正文。
- 拒绝 `improvewriting` 的默认风格偏好询问：当前目标是减少阻断和连续追问。
- 拒绝任何清稿脚本、硬删除列表或大规模禁词表；本轮失败是弱模型执行链路问题，不是缺更多词表。

## 下一轮推荐测试方案

不先改产品文件。先用真实弱模型测试一种“结构化二次修订交互”：

1. 给一篇问题稿和自然语言返工意见。
2. 要求模型先输出三行短单：
   - `保留：` 用户给定事实和动作。
   - `删除：` 未授权字段、流程、格式痕迹。
   - `归位：` 标题、主送/报告对象、正文、落款、日期。
3. 随后输出 `完整改后版本`。
4. verifier 只看“完整改后版本”是否可用，同时记录短单是否帮助减少漏删。

通过口径：

- C1 Claude Code 通知必须删除安装路径、主机名称、用户、报送字段，保留“如有请删除”“未发现也报无”“7月10日15:00通过钉钉发送至技术应用部”。
- C2 OpenClaw 报告必须删除字段标签和后续巡检，把江西出版集团放在主送位置，红星出版社和日期置于末尾。
- 若两轮仍 FAIL，说明 prompt 形态也救不回弱模型，下一步应考虑外部清稿 agent 或更强模型二修，而不是继续改本 skill。
