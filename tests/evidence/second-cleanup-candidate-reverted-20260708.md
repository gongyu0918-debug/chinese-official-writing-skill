# 二次清稿候选回退记录（2026-07-08）

## 目标口径

本轮按更新后的目标执行：弱模型不强求首稿一次成功，但要求在用户用自然语言指出问题后，二次修改能达到可用状态。仍坚持最小修复、软性 prompt、渐进式披露，不把规则堆进 `SKILL.md` 入口，不做一例一修。

## 候选修改

尝试在 `references/review-checklist.md` 增加“二次清稿和返工修订”小节，并在 `SKILL.md` 参考资料表中把 `review-checklist.md` 的加载条件扩展到“二次清稿、修干净、去掉格式痕迹”。

候选规则只在用户明确要求“修干净”“能发”“删掉没说的内容”“去掉格式痕迹”，或已指出具体问题返工时使用；明确不是默认首稿阶段，也不作为中断成稿理由。

同时新增确定性用例：

- `P086`：Claude Code 排查通知二次修改，要求去 Markdown、保留“如有请删除”、不要把 `7月10日15:00发送` 改成 `15:00前`。
- `P087`：OpenClaw 排查报告清稿，要求不补排查范围、检查方法、经办字段、后续管理动作。
- `P088`：二次清稿后主送、正文、落款、日期回到可发文位置。

## 真实写稿与二次修改复测

### 第一组：6 个真实 prompt

弱模型首稿覆盖电脑采购情况说明、微博情况说明、Claude Code URL 改通知、气象 URL 改通知、世界杯期间通知、OpenClaw 排查报告。

独立 verifier 判定：

- 弱模型首稿：W3 为 FAIL，其余多为 WARN。
- 弱模型二次修改：多数正文进入可用区间，但 6/6 仍残留 Markdown 标题加粗，W6 报送对象位置不规范。
- 强模型首稿：整体更好，但仍在 URL 通知、气象通知和 OpenClaw 报告中补入未给的执行链条或后续管理动作。

据此才进入候选修复，重点放在二次清稿链路，而不是继续加首稿默认阻断。

### 第二组：候选后 4 个首稿 + 二次修改

弱模型首稿仍没有明显改善：

- W2 Claude Code 通知仍新增工作电脑、测试环境、临时环境、记录版本、主机名称、用户、安装路径、报送字段，并把时间写成 `15:00 前`。
- W3 世界杯通知仍新增值班、签到、核验、异常处理、资金异常、处置上报等执行链条。
- W4 OpenClaw 报告仍新增核查范围、安装记录、残留文件、开机自启项、常驻进程、后续巡检和月度复核。

用户自然语言指出问题后二次修改有所改善：能删掉多数未授权链条，能修正 Claude Code 通知里的“如有请删除”和定点时间，但仍有共性残留：

- 落款和日期压在正文句末。
- 主送或报告对象没有放在正文前可发文位置。
- 个别样本仍有轻微事实外扩。

独立 verifier 结论：`WARN，不建议直接判定本轮修复完全达标`，建议若继续要补“可发文首尾骨架分行输出”。

### 第三组：补首尾骨架后 targeted 复测

新增“清稿后按可发文首尾骨架分行输出”后，弱模型对 C1/C2 二次清稿仍未达标：

- C1 仍保留 `标题：` 标签，仍保留用户明确要求删除的主机名称、用户、安装路径、报送字段。
- C2 仍保留 `标题：`、`报 告 人（发送方）`、`报 告 对 象` 等字段痕迹，未把江西出版集团处理成自然主送位置。

独立 verifier 结论：`WARN`，弱模型能删除多数用户明确不要的内容，但字段式痕迹和主送位置仍不稳定；`不建议保留本轮修改作为发布候选`。

## 下一步建议

- 不要继续往 `SKILL.md` 或 `review-checklist.md` 堆相邻清稿规则；弱模型已经显示会选择性忽略。
- 如果继续做二次修订能力，应优先测试更明确的外部清稿链路或更结构化的二次修订交互，但不能写成默认首稿阻断。
- 对弱模型可放宽为“首稿基本可用，二次指出问题后可用”，但本轮 targeted 复测没有达到这个口径。
- 下一轮必须继续使用真实用户式 prompt 和独立 verifier；确定性消融只能作为工程回归证据。
