# 社区经验增强专项测试 Prompt

本文件为脱敏测试清单，不包含用户材料、真实单位名称、真实金额或内部项目事实。

## 办理要素收集

1. 起草一份通知，已知事项为报送季度材料，未给出截止时间和反馈渠道。
2. 起草一份请示，事项为申请启动专项工作，已知背景但未给出请批事项表述。
3. 起草一份报告，要求汇报阶段进展，但用户输入中夹带“请批准下一步经费”。
4. 起草一份函，要求平行单位提供接口联调条件，未给出联系人。
5. 起草一份批复，来文请求同意试点，需明确同意范围和后续要求。
6. 起草一份会议纪要，会议形成三项责任分工和两个未决事项。
7. 起草一份工作总结，已有成果数据但缺少问题和下一步安排。
8. 起草一份调研报告，已有现象和建议但缺少样本来源。
9. 起草一份建设方案，用户只给了项目名称和建设目标。
10. 起草一份审查材料，需区分通过项、补充项和整改项。

## 文种功能边界

1. 用户要求“写报告请领导批钱”，检查是否拆分为报告和请示。
2. 用户要求“给同级单位下通知”，检查是否改为函或商请。
3. 用户要求“批复对方来函”，检查是否判断上下级关系。
4. 用户要求“会议纪要写成新闻稿”，检查是否回到议定事项。
5. 用户要求“公示但没有反馈渠道”，检查是否补出公示要素。
6. 用户要求“公告面向内部部门”，检查是否改为通知或公示。
7. 用户要求“申请但没有申请事项”，检查是否先明确请求。
8. 用户要求“说明原因并请求批准”，检查是否拆分文种。
9. 用户要求“方案只写背景”，检查是否补齐任务、责任、验收。
10. 用户要求“审查材料只写基本同意”，检查是否补出依据和条件。

## 论证链条

1. 请示：请批事项 -> 依据 -> 必要性 -> 资金安排 -> 风险控制 -> 请批语。
2. 报告：背景 -> 进展 -> 成效 -> 问题 -> 下一步。
3. 通知：事项 -> 对象 -> 材料 -> 时限 -> 联系方式。
4. 函：商请事项 -> 配合内容 -> 回复时限 -> 联系方式。
5. 建设方案：需求 -> 差距 -> 建设内容 -> 资源成本 -> 成效。
6. 实施方案：目标 -> 阶段 -> 任务 -> 责任 -> 验收。
7. 调研报告：事实 -> 问题 -> 原因 -> 建议。
8. 审查材料：对象 -> 依据 -> 发现 -> 风险 -> 结论。
9. 算力租赁方案：需求来源 -> Token 测算 -> 成本比较 -> SLA/并发 -> 安全验收。
10. 云端部署成本说明：当前费用 -> 增长因素 -> 对比路径 -> 风险 -> 建议。

## 成稿复核层

1. 用户固定文章标题，只检查正文是否贴题，不改主标题。
2. 用户固定一级标题和二级标题，只改正文偏题部分。
3. Agent 生成的二级标题过度承诺，检查是否降为正文可支撑的标题。
4. 相邻两段重复说明同一成本事项，检查是否合并。
5. 上一段已写建设必要性，下一段换词重复必要性，检查是否删减。
6. 文中出现“我的思路”“接下来我会”，检查是否删除思考过程泄露。
7. 文中出现半角逗号、半角冒号和中文正文混用，检查是否改为中文标点。
8. 文中数字出现空格和千位分隔符，检查是否符合中文正式材料习惯。
9. 文中频繁使用 `1. 2. 3.` 和项目符号，检查是否改为中文条款或自然段。
10. 文中使用 Emoji 或装饰性符号，检查是否删除。
