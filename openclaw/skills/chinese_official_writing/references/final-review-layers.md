# Final Review Layers

Use this reference after drafting and before delivery. It is designed for generated formal materials where sections may have been produced in batches.

## Three-Layer Review Policy

- Hard boundaries come first: document genre, unsupported facts, explicit user requirements, sensitive information, and process leakage. These issues must be fixed before delivery.
- Quality suggestions come second: low-AI flavor, repetition, loose structure, list fatigue, heading polish, table overuse, and format noise. These are advisory unless they hide a hard-boundary problem.
- Scenario references come last: load genre, handling, argument-chain, AI-compute, DOCX, or task-specific scenario references only when the task needs them.

## Title and Heading Review

- Do not change the main article title by default.
- If the user fixed headings, keep those headings and check whether each section's body serves the heading.
- If headings were generated by the agent, consider polishing headings that are vague, duplicated, too technical, or off-topic.
- Check large headings and small headings separately. A small heading should not introduce a claim that the following paragraph does not support.
- If a section drifts away from its heading, revise the body first. Change the heading only when the body is correct and the heading was not user-specified.
- Avoid headings that merely repeat abstract words such as `总体情况`, `相关说明`, `保障措施`, unless the document template requires them.

## Repeated-Matter Review

- Compare each paragraph with the previous paragraph and the previous section.
- If the same matter appears again, keep it only when the new paragraph adds one of these: new data, new reason, new decision, new responsibility, new risk, new deadline, or new acceptance requirement.
- Consider merging paragraphs that only restate the same claim with synonyms.
- In project documents, watch for repeated claims across `建设必要性`, `需求测算`, `建设方案`, `建设成效`, and `安全保障`.
- In cost sections, check repeated numbers and keep later occurrences when needed for comparison or conclusion.
- In AI compute materials, prefer placing demand, Token/resource, cost, SLA, security, and acceptance in their strongest section; cross-reference lightly when useful.

## Anti-AI Review

- Viewpoint consistency: the text must speak from the issuing unit, reporting unit, project unit, or supervising unit selected for the document.
- Side-commentary: check statements that tell the reader how the document is written and convert them to正文 when they are not part of the user's requested form.
- Thought leakage: remove model identity, hidden reasoning, prompts, recording instructions, user instructions, and drafting process. This is a hard-boundary issue.
- Paired frames: consider reducing decorative `不是……而是……`, `不仅……还……`, `既……又……`, `一方面……另一方面……`. Keep them when they carry a real decision distinction.
- List fatigue: frequent `一是、二是、三是` or `1. 2. 3.` is a style risk, not an automatic failure. Adjust only when it makes the draft mechanical or when the genre does not expect 条款.

## Format Review

- Chinese prose generally uses full-width punctuation. Keep half-width punctuation for URLs, code, formulas, model names, file paths, and English abbreviations.
- Confirm number style: check unnecessary spaces around numbers and thousands separators unless the source template uses them.
- Preserve first-line indentation, heading levels, fonts, and paragraph spacing in DOCX.
- Check table overuse. Tables should carry structured data, schedules, budgets, responsibility lists, or comparisons; prose should carry the argument.
- Remove Emoji and decorative bullets from formal prose unless the user is preparing an informal slide, poster, or product note rather than official prose.
- Use `scripts/prose_lint.py --format --structure` for a risk scan when reviewing a plain text, Markdown, or extracted DOCX draft.
